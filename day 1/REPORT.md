# День 1. Rules — отчёт об итерации файла правил

## Что делалось

1. Проанализирован house style прошлого марафона (`../AI-Advent-Challenge-8`, day 1–35):
   стек, структура дней, реальные файлы (`Config.kt`, `DeepSeekClient.kt`, `HttpApi.kt`,
   `TaskState.kt`).
2. Написан корневой **CLAUDE.md v1**: стек, архитектура, структура папок, naming,
   паттерны, 4 примера хорошего кода из проекта, 5 антипаттернов, шаблон файла.
   Использованы все четыре уровня: глобальный конфиг (`~/.claude/CLAUDE.md`),
   локальный (`CLAUDE.md` в корне), субагенты (`day-builder`, `day-reviewer`),
   скиллы (`new-day`, `verify-day`).
3. **Боевая задача одним промптом** (без уточнений) субагенту `day-builder`:

   > День 1. Сделай HTTP-сервис «мотиватор»: GET /healthz; POST /v1/motivate —
   > {"task": "..."} → мотивационная фраза через DeepSeek; GET /v1/history —
   > последние 10 запросов (в памяти). Порт 8080.

4. **Генерация №1** (правила v1) → ревью субагентом `day-reviewer` → пробелы правил
   → **CLAUDE.md v2** → та же задача тем же промптом с нуля → **генерация №2** → дифф.

Артефакты: `report/gen1/` — исходники первой генерации; текущий `day 1/` — вторая;
`report/gen1-vs-gen2.diff` — полный дифф (542 строки).

## Генерация №1: результат

Скомпилировалось: **да** (одна самопочинка по ходу: `ArrayDeque.reversed()` на JDK 21
резолвится в `SequencedCollection` — агент сам заменил на `toList().asReversed()`).
Живой прогон: **да** (healthz, motivate с реальным DeepSeek, history, 400/405).
Стиль: вердикт ревью **PASS**, но с замечаниями:

- KDoc обещал порядок обороны `405 → 413 → 400`, код делал `405 → 400 → 413`;
- тело запроса читалось безлимитным `readBytes()` до проверки размера;
- нет `demo.sh` и секции «Чеклист задания» в README (есть во всех днях 21–35 образца).

## Что не учёл файл правил v1 (найдено ревью)

1. `demo.sh` — обязателен в house style, в правилах не упоминался.
2. Каталог `output/` для генерируемых артефактов + `_run.log` через `tee`.
3. Секция `## Чеклист задания` и общий шаблон README.
4. **Внутреннее противоречие**: правило говорило «ошибки `{error, message}`»,
   а цитируемый в тех же правилах пример из day 30 использует вложенный
   `{"error": {"code", "message"}}`. Генератор угадал правильно, но по наитию.
5. Access-лог через обёртку `handle()` — паттерн образца, не был зафиксирован.
6. Порядок обороны был описан неточно (без «413 по Content-Length ДО чтения тела»).
7. Конвенция портов 8000 + N.
8. Политика тестов (по умолчанию — нет; если нужны — kotlin-test + junit 5.11.3).
9. `.gitignore` не был в списке обязательных файлов.
10. Явное «graceful shutdown не требуется» (чтобы не изобретались shutdown-hook'и).

Все 10 пунктов внесены в **CLAUDE.md v2**.

## Дифф генераций (главное)

| Область | Gen №1 (v1) | Gen №2 (v2) |
|---|---|---|
| `demo.sh` | нет | есть, нумерованный curl-сценарий |
| `run.sh` | `exec gradlew run` | `… \| tee output/_run.log` |
| `.gitignore` | без output | + `output/_run.log` |
| Оборона `/v1/motivate` | 405 → 400 → 413, `readBytes()` без лимита | 405 → 413 по `Content-Length` **до чтения** → `readNBytes(потолок+1)` → 400 → 502; KDoc совпадает с кодом |
| README | таблица API, без чеклиста | ASCII-диаграмма потока, таблица env, `## Чеклист задания` |
| Порт | 8080 молча | 8080 с явным комментарием «осознанное отступление от конвенции 8000+N по условию дня» |
| Access-лог | без статуса ответа | одна строка со статусом, как в правиле |
| Config | порт/лимит task | + `MAX_BODY_BYTES`, `HISTORY_SIZE` — всё, что стало правилом, стало конфигом |

Обе генерации: компилируются, живой прогон пройден, стек/структура/naming без нарушений.

## Что больше всего повлияло на качество

1. **Примеры реального кода в правилах** — самый сильный фактор: Config, клиент
   и HTTP-слой обеих генераций почти дословно повторяют образцы. Что показано кодом —
   воспроизводится; что описано словами — интерпретируется.
2. **Явный порядок обороны HTTP с уточнением «до чтения тела»** — единственное
   *поведенческое* (не косметическое) улучшение gen2; из абстрактного «лимиты →
   валидация» v1 генератор сделал уязвимую версию, из конкретного правила v2 — верную.
3. **Чек-листы обязательных файлов** (demo.sh, .gitignore, output/) — бинарные правила
   выполняются в 100% случаев; всё «самоочевидное из образца» без фиксации теряется.
4. **Устранение противоречий**: правило, конфликтующее с примером в том же файле, —
   лотерея; в v1 повезло, рассчитывать на это нельзя.
5. **Ревью отдельным агентом против образца** — именно day-reviewer нашёл 10 пробелов
   правил, которые не видны при написании правил «из головы».

## Кандидаты в v3 (замечено на gen2)

- Gen2 заменил sealed `MotivationResult` из gen1 на исключение + try/catch в HttpApi —
  паттерн №3 стоит уточнить: «клиент бросает (`check`), доменная операция с ожидаемыми
  исходами возвращает sealed». Словесной формулировки оказалось мало — нужен пример
  именно доменного слоя, а не state machine.
- `/v1/history` в gen2 собирает JSON конкатенацией строк (`sendRaw` + `buildString`) —
  зафиксировать «JSON только через kotlinx.serialization, руками строки не клеить».
