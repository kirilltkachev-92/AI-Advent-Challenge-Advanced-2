# День 2. Profiles — отчёт

## Процесс

1. Research: изучен day 1 — найден **настоящий** (не подсаженный) баг для теста
   Bug Fix: `HistoryStore.add()` при `HISTORY_SIZE=0` делает `removeFirst()` на
   пустом деке → `NoSuchElementException` → 500 на каждом успешном `/v1/motivate`.
   Баг воспроизвели в живом окружении (`HISTORY_SIZE=0` в `.env`).
2. Написаны профили v1: `bug-fix`, `research`, `day-docs` (третий — под частую
   задачу марафона: синхронизация доков дня с кодом).
3. Каждый профиль проверен на реальной задаче: **один промпт, без уточнений**.
4. По итогам прогонов профили доработаны до v2. Повторный прогон не потребовался —
   все три дали рабочий результат с первой попытки; доработки закрывают найденные
   слабые места, а не провалы.

## Что доработано в профилях после первой попытки (v1 → v2)

**bug-fix:**
- + «если баг связан с конфигом/env — прогнать в регрессии ОБА режима (значение
  из бага и дефолт)» — агент сделал это по своей инициативе; поведение, которое
  спасло бы от полуфикса, закреплено правилом;
- + «восстановить окружение после себя: остановить сервисы, убрать временные
  env-переопределения, `.env` пользователя не менять» — в v1 не было ни слова,
  агенту повезло сделать правильно.

**research:**
- + «прочитать README/REPORT дня ПРЕЖДЕ чем утверждать „не задокументировано“» —
  единственная реальная слабость первой попытки: агент честно написал, что README
  не читал, рассуждая об «условии дня»;
- + секция ответа «Где не смотрел» — границы исследования теперь часть контракта.

**day-docs:**
- + секция ответа «Замечено в коде (не правил)» — агент нашёл косметический баг
  (баннер «последние 0») и втиснул его в «Конфликты»; теперь у таких находок
  есть штатное место, откуда они уходят в bug-fix;
- + явно разрешён короткий живой прогон сервиса с остановкой после — v1 умалчивал,
  агент рискнул (и правильно), теперь это часть контракта.

## Наблюдения

1. Жёсткий формат ответа в профиле — главный рычаг: все три агента выдали ровно
   заданную структуру, ответы сравнимы между прогонами.
2. Ограничение инструментов (research без Write/Edit) надёжнее текстового запрета:
   «не должен менять файлы» подкреплено физически.
3. Порядок «воспроизведи → локализуй → чини причину → регрессия» в bug-fix дал
   фикс причины (guard в HistoryStore), а не симптома (try/catch в HttpApi).
4. Требование «файл:строка» в research исключает ответы-эссе: каждое утверждение
   проверяемо.
5. Профили усиливают друг друга: research нашёл непокрытые ветви demo.sh,
   day-docs нашёл кандидата для bug-fix — конвейер из профилей складывается сам.

## Ограничение сессии

Новые типы субагентов подхватываются Claude Code при старте сессии, поэтому в
этой сессии профили запускались через catch-all агента с текстом профиля как
системной инструкцией — содержательно это тот же тест. Со следующей сессии
`bug-fix` / `research` / `day-docs` доступны как нативные subagent_type.
