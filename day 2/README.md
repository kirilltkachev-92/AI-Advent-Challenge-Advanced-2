# День 2. Profiles

Три профиля агентного режима для этого репозитория, оформленные как субагенты
Claude Code (`.claude/agents/`):

| Профиль | Файл | Назначение |
|---|---|---|
| **Bug Fix** | `.claude/agents/bug-fix.md` | Получает симптом → воспроизводит → находит причину (файл:строка) → минимальный фикс → проверяет, что не сломал остальное |
| **Research** | `.claude/agents/research.md` | Вопрос по кодовой базе → read-only исследование → структурированный ответ со ссылками файл:строка |
| **Docs** (усиление) | `.claude/agents/day-docs.md` | Частая задача марафона: синхронизация README/REPORT дня с кодом по шаблону из CLAUDE.md |

Каждый профиль описывает: системную инструкцию, что агент ДОЛЖЕН делать,
что НЕ ДОЛЖЕН, и жёсткий формат ответа. Research ограничен инструментами
Read/Bash/Glob/Grep (без Write/Edit) — запрет на изменения обеспечен не только
текстом, но и списком tools.

## Как использовать

Из корня репозитория (новая сессия Claude Code подхватывает `.claude/agents/`):

```
Задача для субагента bug-fix: <симптом бага>
Задача для субагента research: <вопрос по кодовой базе>
Задача для субагента day-docs: day N — <что изменилось>
```

## Проверка на реальных задачах (по одному промпту, без переписок)

- **bug-fix** — реальный баг day 1: `HISTORY_SIZE=0` ронял каждый `/v1/motivate`
  в 500 (`removeFirst()` на пустом деке в `HistoryStore.add`). Найден, починен
  минимальным диффом, регрессия прогнана. → `logs/bug-fix-run.md`
- **research** — вопрос «как устроена оборона HTTP в day 1 и какие ветви не
  покрыты demo.sh». Полная карта обороны + 8 непокрытых ветвей. → `logs/research-run.md`
- **day-docs** — после фикса синхронизировал README day 1 с новым поведением,
  сверил env-таблицу с Config.kt построчно. → `logs/day-docs-run.md`

Все три — рабочий результат с первого запуска. Что доработано в профилях после
первой попытки — `REPORT.md`.
